Transaction Flow: Client submits transaction → Routed to relevant shards (clusters) → Each cluster runs Linear PBFT consensus (Pre-prepare → Prepare → Commit phases) → For cross-shard transactions, 2PC protocol coordinates (Prepare-to-Lock → Vote YES/NO → Commit/Abort) → Pessimistic locking prevents double-spending → Transaction finalized across all shards or fully aborted

Network Scale:
12 Servers (3 Clusters × 4 Nodes)
  • Linear PBFT Consensus: Custom implementation with O(n) complexity tolerating up to f Byzantine faults per cluster. Three-phase protocol (Pre-prepare, Prepare, Commit) ensures all honest nodes agree on transaction order.
  • Atomic Cross-Shard Transactions: Two-Phase Commit protocol coordinates transactions across multiple shards. Ensures atomicity—transactions either fully complete or fully abort across all involved clusters.
  • Double-Spending Prevention: Pessimistic locking mechanism during Prepare-to-Lock phase. Accounts are locked before transaction execution, preventing simultaneous conflicting transactions from spending the same funds twice.
  • Sharded Architecture: Network divided into 3 independent clusters for parallel transaction processing. Demonstrates blockchain scalability through horizontal partitioning of data and workload distribution.
